# direction-explore (stage 0)

plan·build·review 앞에 오는 **방향 탐색** 단계다. 여기는 **오라클이 셋 중에서도 가장 없다 — 초안조차 없다.** 그래서 엔진은 friction(수렴)이 아니라 그 반대 짝 **발산**: 서로 다른 접근을 독립 생성 → judge-panel로 비교 → 종합해 초안 하나. (오라클 가용성이 오케스트레이션 모양을 정한다.)

## 절대 규칙
- **오라클 0.** 발산도 판정도 결국 LLM 판단이다. **수렴(후보들이 비슷)을 정답으로 착각하지 마라.** repo가 있으면 그게 유일한 ground-truth 근사(feasibility·기존 정합) — 적극 활용.
- **decorrelation이 이 단계의 핵심 방어다.** 생성기들은 서로의 출력을 *안 본다*. 같은 모델이라도 stance를 갈라 anchoring(첫 아이디어 고착)을 깬다.
- **이건 초안 *생성*이지 *검증*이 아니다.** 산출물(초안)은 반드시 `plan-friction-loop`로 넘겨 압박해야 한다. 여기서 끝내지 마라.

## 적용 판단 (먼저)
- **솔루션 공간이 넓고 갈릴 때만** 써라 — 아키텍처 선택, 접근 A/B/C, "어떻게 가야 할지 모름". 발산이 단일 모델 첫 아이디어 고착을 깬다.
- **방향이 거의 정해졌거나 좁으면 쓰지 마라**(과잉). 그냥 초안 한 문단 잡아 바로 `plan-friction-loop`로. 코드 기반 "어디 손대지/뭐가 닿지"(영향 범위 파악)면 explore보다 해당 코드의 흐름을 먼저 읽는 게 맞다.
- **이게 explore의 규모 적응이다**: explore는 *방향 불확실성*이 높을 때만 켜지는 단계다. auto가 profile을 micro/small로 줬거나(state.md) 방향이 정해졌으면 이 단계를 건너뛰고 plan(또는 build)으로. 발산 stance 수도 규모 따라(기본 3, 공간 좁으면 2). ⚠️ explore는 **부트스트랩이라 incoming handoff가 없다** — 규모는 요청 형용사가 아니라 repo/코드 흐름 읽기(위험 표면·기존 정합)로 grounded하게. 그리고 judge의 "갈림 vs 공간 좁음" 판정을 `handoff.explore→plan: 갈린 가정`에 적어 *다음 단계가 규모를 읽을* 신호로 남겨라.

## 입력
- **goal**: 이루려는 것(한 줄~문단). plan 아님 — "무엇을" 만 있으면 됨.
- **repo (선택)**: 주어지면 생성기·judge가 기존 코드·관습을 Read/Grep으로 보고 현실에 묶는다(허구 설계 방지). 이 단계의 유일한 오라클 근사.

## 단계 상태 (lifecycle 척추)
이 스킬은 파이프라인의 **stage 0**이다. 완료 시 대상 repo의 `.cidd/state.md`에 등록(스키마 README): `active: <slug>`, `explore: done → .cidd/explorations/<slug>.md`, `stage: plan`(다음 준비됨), `handoff.explore→plan`(선택한 방향 + 갈린 핵심 가정 = friction이 압박할 곳). 초안은 plan-friction-loop가 읽어 `.cidd/plans/<slug>.md`로 다듬는다. ⚠️ 단계 끝은 *사람 게이트*다 — 사용자가 방향을 거부하면 stage를 강제로 plan으로 밀지 마라. `.cidd/explorations/<slug>.md`의 후보들은 **재진입 메뉴**(나중에 다른 후보 선택 가능), 거부된 방향은 `rejected.explore`에 기록해 재실행이 피하게 하라. **끝나면 `AskUserQuestion`으로 결정 메뉴(accept/refine/back/pause/abandon)를 띄우고 고른 전이를 state.md에 자동 적용한다**(README "단계 끝 = 결정 메뉴"). 방향이 별로면 refine(다른 stance로 재발산) / abandon.

## 절차

### 1. stance 도출 (다양성의 엔진)
서로 다른 각도 **3개(기본)**를 goal에 맞춰 고른다 — 같은 각도 N개는 의미 없다. 후보 stance:
`MVP-우선`(최소로 동작) · `리스크-우선`(실패·엣지부터 설계) · `기존정합-우선`(코드베이스 관습 따라) · `단순성-우선`(적은 코드·의존) · `확장성-우선`(미래 변경 대비). goal·repo에 맞는 저상관 3개.

### 2. 발산 (병렬, 독립)
`approach-generator`를 stance마다 하나씩 한 메시지에서 동시 `Agent` 호출. 각자 goal + *자기 stance* + (있으면) repo 경로. **서로의 출력을 안 본다.** 각 생성기는 자기 stance를 끝까지 밀어 한 접근안을 낸다(균형 잡힌 답 금지 — 그건 다른 후보 몫).

### 3. 판정 (judge-panel)
`approach-judge`에 후보 전부 넘긴다. 기준: 목표 적합 / 단순성 / 리스크·되돌리기 / 노력 / 기존 정합(repo로 확인). 반환: ranking + winner + 패자에서 **접목할 부분** + 갈린 핵심 가정.
- ⚠️ judge의 판정은 검증이 아니라 판단이다. 후보들이 진짜 비슷하면 judge가 "공간 좁음"이라 말하게 두고 — 그럼 발산은 무의미했던 것, 그냥 초안 잡고 4로.

### 4. 종합 (네가)
승자를 뼈대로, 패자들의 좋은 아이디어를 접목해 **초안 plan 하나**를 쓴다. **갈린 가정·미해결 결정을 명시**(friction이 물 지점). "둘 다 한다"로 얼버무리지 마라 — 방향을 *정해라*, 근거와 함께.

## 출력
- `.cidd/explorations/<slug>.md`: 후보 N개 요약 + 판정표(ranking·근거) + 선택 이유 + **초안 plan** + 갈린 가정.
- 인라인 — 결과물 우선, 과정은 파일로. 순서: 리드(1줄) → **결과물 = 선택한 방향 + 초안 plan**(선택 방향 한 줄 + 초안 핵심: 무엇을·갈린 가정 + 후보 한눈 비교표) → 갈린 핵심 가정이 사람이 정할 fork면 결정 포인트 → 파일 포인터. ⚠️ 후보별 raw·판정 근거 전체는 explorations 파일로(→ 경로 한 줄). 게이트: 평이어·장식기호 금지·빈 섹션 생략.
- **다음**: 초안을 `/cidd:plan-friction-loop`로 넘겨라(state.md가 stage:plan으로 가리킴). 탐색은 검증이 아니다 — 거기서 압박받아야 한다.

## 비용·정직성
- 생성·판정은 *설계*라 capable 모델이 필요(에이전트 모델 미고정 = 세션 상속). 비용 크면 N=2로 줄이거나 모델 낮춰라. 발산이 값있는 *넓은* 결정에만 이 스킬을 써라.
- 제어흐름은 코드가 아니라 지시다 — 독립 생성(서로 안 봄)·judge·종합 순서를 문자대로 지켜라. 발산 건너뛰고 바로 초안 쓰면 이 스킬을 쓸 이유가 없다.
